# How Migration Acceleration Works

## Purpose
This framework helps SEs qualify opportunities and gather the minimum decision data needed to move an opportunity from Uncommitted to Committed in the MCEM process.

This is a presales acceleration layer. It does not replace CSA, Factory, or delivery engineering.

## The 6-Stage Flow

### Stage 1: Discovery Capture (30 minutes)
Run one structured call and capture commit-critical data in:
- `accounts/00-account-template/01-discovery-answers.md`

Focus on:
- Business outcome and decision path
- Wave-1 high-level scope
- Readiness RAG and key dependencies
- Top risks and unknowns

### Stage 2: Commit Confidence Check (10 minutes)
Review whether data quality is sufficient for a stage movement recommendation.

Use:
- `accounts/00-account-template/06-commit-gate-check.md`

### Stage 3: Define 2-Week Plan (10 minutes)
If confidence is not yet sufficient, define owner/date actions to close gaps.

Focus on:
- Unknowns that block commitment
- Risks without ownership
- Missing commit artifacts

### Stage 4: Produce Handoff Artifacts (10 minutes)
Generate both outputs:
- `templates/01-loop-readiness-output-template.md`
- `templates/02-factory-nomination-pack-template.md`

### Stage 5: Share and Track (5 minutes)
Share outputs with CSU/Factory and track outcome in:
- `docs/04-pilot-kickoff-and-tracker.md`
- `templates/03-feedback-form-template.md`

### Stage 6: Stage Recommendation
Record recommendation and sign-off path.

Outcome:
- Move to Committed, or
- Stay Uncommitted with dated closure actions

## What Good Looks Like
An account is commit-ready when all are true:
- Business outcome, urgency, and approval path are explicit
- Wave-1 scope is clear at presales level
- Top risks have owner and target date
- Unknowns that affect commitment have closure dates
- Required commit artifacts are in place

## What This Is Not
- Not low-level delivery design
- Not full migration planning
- Not detailed implementation architecture

## What This Solves
Most delays happen before delivery starts because commitment decisions are made with incomplete data. This workflow improves decision confidence and handoff quality so delivery can start faster with less rework.
